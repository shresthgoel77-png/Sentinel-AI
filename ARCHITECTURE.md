# Sentinel AI Architecture Documentation

## 1. System Architecture & Vision

Sentinel AI began as a client-side, browser-based security scanner designed for conceptual demonstrations of AI guardrails. It has recently evolved into a production-grade, live runtime AI Gateway. Operating as an interception proxy between client applications and downstream LLM API providers (such as OpenAI and Anthropic), Sentinel AI provides real-time security enforcement, preventing prompt injections, jailbreaks, and data leaks before they reach the model.

**Tech Stack:**
*   **Backend:** Python via the FastAPI asynchronous web framework.
*   **Database Layers:** PostgreSQL (managed by SQLAlchemy ORM and Alembic migrations) for relational data such as Tenants, API Keys, Policies, and Audit Logs. Upstash Redis handles high-concurrency state management for the realtime scanning interface.
*   **Security Engine:** LangChain and LangGraph power a stateful, zero-trust evaluation pipeline utilizing Gemini flash models as the core detection heuristics brain.
*   **Frontend:** React (Vite) and Tailwind CSS with interactive security dashboards connecting directly to the live backend.

## 2. Request Lifecycle (Ingress to Egress)

When a client application sends a prompt request to Sentinel AI's core gateway, the request follows a strict, sequential security lifecycle:

1.  **Ingress:** The request arrives at the OpenAI-compatible endpoint `POST /v1/chat/completions`.
2.  **Authentication (`HTTPBearer`):** The provided API key (`sk_sentinel_...`) is hashed and validated against the PostgreSQL `api_keys` table. The associated Tenant and Policy constraints are instantly loaded.
3.  **Sanitization Middleware:** The prompt string is compiled and passed through the `DocumentSanitizer`, which scans for direct hard-coded indicators of compromise (IoCs) and PII anomalies.
4.  **Security Analysis (LangGraph):** The sanitized text is dispatched to the `app_graph` execution pool via asyncio. The stateful Gemini-powered graph actively scores the intent of the prompt (checking for jailbreaks and manipulation).
5.  **Risk Evaluation:** The resulting `risk_score` is evaluated against the tenant policy (`max_risk_score`, typically defaulting around 80). If the score exceeds the threshold, the gateway immediately cuts the connection, returning a 4XX security violation, and writes a blocked audit entry asynchronously.
6.  **Provider Abstraction (Routing):** Valid requests are dynamically routed based on the requested model name (e.g., `gpt-4` uses `OpenAIProvider`, `claude-3` uses `AnthropicProvider`).
7.  **Output Scan (Egress Redaction):** The response payload generated by the LLM is captured and run through a regex redactor to obscure sensitive data (emails, API credentials) before egressing back to the client.
8.  **DB Logging (Asynchronous):** All request telemetry (tokens used, latency, threat signals, provider routing) is safely handed off to FastAPI `BackgroundTasks` for insertion into the `audit_logs` table, ensuring zero latency penalty to the client.

## 3. Backend Breakdown

The backend codebase (`backend/`) is structurally modular, isolating API orchestration, database layers, ML provider integrations, and security engines.

*   **`main.py`:** The nucleus of the FastAPI application. It hosts the startup lifecycle (Redis initialization), coordinates CORS for the frontend, manages the legacy analysis endpoints (`/api/analyze-document`), and most importantly, defines the OpenAI-compatible gateway interceptor (`/v1/chat/completions`) and its asynchronous background audit log dispatchers.
*   **`database/`:** 
    *   **`models.py`:** Defines the SQLAlchemy ORM schema. Contains the relational definitions for multi-tenancy: `Tenant` (organization accounts), `APIKey` (associated keys), `Policy` (tenant-specific threshold configs), and `AuditLog` (transactional history).
    *   **Alembic:** Utilizes generated migration scripts to handle schema synchronization against the PostgreSQL instance.
*   **`providers/`:** 
    *   **`base.py`:** Mandates an abstract standard interface (`generate_completion`) for all provider integrations.
    *   **`openai.py` & `anthropic.py`:** Implement the physical downstream API calls utilizing the native `AsyncOpenAI` and `AsyncAnthropic` SDKs. Crucially, the Anthropic provider implements response normalizers to map Claude's output schema back into the OpenAI `chat.completion` spec structure, ensuring client agnostic compatibility.
*   **Security Pipeline:**
    *   **`langgraph_engine.py`:** Serves as the synchronous/asynchronous inter-process bridge. It maps gateway and backend payloads into the rigid `ThreatState` TypedDict standard expected by the graph.
    *   **`graph.py`:** The physical state machine (DAG). Uses Google Gemini (`gemini-2.5-flash`) as the zero-trust classification intent brain. Nodes progress through heuristic scanning, semantic document scanning, quarantine routing, authorization routing, and finally, LLM explanation generation for flagged segments.

## 4. Frontend Breakdown

The client application (`src/`) has transitioned from a visually appealing but logically hollow mockup to a fully integrated terminal client connecting directly to the new python Gateway.

*   **`DataLeakDemo.tsx`:** Previously relying on local synchronous `setTimeout` delays to simulate scanning, this component was heavily refactored. The `runAnalysis` executor was re-written to trigger a live un-mocked `fetch` payload against `http://localhost:8000/v1/chat/completions`. It explicitly sends a `gpt-4` model intent, authenticates utilizing the test key `sk_sentinel_test_123`, and seamlessly maps the actual resulting output attributes (like `_sentinel_trace.langgraph_risk` and injected `threat_signals`) directly into the robust TypeScript `<RiskMeter>` and `<ShieldBadge>` React state variables.
*   **`Hero.tsx`:** Acts as the primary funnel. The core "Try Live Gateway" Call to Action (CTA) button was wired dynamically. The legacy no-op href was replaced with an `onClick` event listener engineered to programmatically initiate a smooth viewport scroll to the mounted live interactive dashboard section (`#live-gateway`), followed by an immediate DOM focus command placing the user cursor directly into the gateway test input.
